Surgical Scheduling Dashboard

A dashboard for optimizing surgery schedules and enhancing the process of information processing for surgical booking

Overview

This case study documents my work on the Surgical OR Scheduling System for ORCHID Analytics, a dashboard designed for a children's hospital in North America to help administrators model "what-if" scenarios for surgical capacity planning.

My Journey with This Project

Phase 1: Research & Initial Design (2021-2022)

As the sole UX researcher and designer for 17 months, I conducted comprehensive research across 8 user groups and created initial Figma designs for the dashboard.

Phase 2: Feedback & Departure (2022)

Through user testing, I identified key improvements needed. However, the company ran out of funding before I could implement these changes.

Phase 3: AI-Assisted Redesign (2024-2025)

I revisited this project, using AI to implement the feedback and suggestions I had documented, creating a fully functional redesign.

What This Case Study Demonstrates: How comprehensive UX research drives design decisions, how feedback identifies areas for improvement, and how AI tools can accelerate implementation‚Äîtransforming documented insights into a functional prototype that would have taken months with traditional development.

The Problem Space

Healthcare Context

Hospital operating rooms are complex environments where 14-16 OR rooms operate daily across multiple surgical services. Surgeons, staff, and patients must be coordinated efficiently through block schedules that allocate time blocks (6.5-10.5 hours) to different surgical departments. Effective waitlist management directly impacts patient outcomes and hospital revenue.

Business Challenge

Hospital administrators needed a system to:

  • Visualize current OR block allocations across services (Plastics, Ortho, ENT, etc.)
  • Model "what-if" scenarios (e.g., "What if we add weekend OR hours?")
  • Track KPIs: Waitlist Reduction, Overtime Hours, Bed Occupancy, QBP Funding, Staffing Requirements
  • Drill down from high-level OR grids to surgeon schedules to individual patient allocations

I spent several months embedded in the hospital's surgical scheduling ecosystem. Surgical scheduling involves multiple decision-makers with different needs and time horizons‚ÄîI needed to understand not just what they do, but why they make certain decisions and when they need specific information.

üí° Key Discoveries

Key User Insights

Complex Decision Matrix: "Division Managers juggle multiple factors: surgeon availability, equipment resources, post-op capacity, and HR constraints - all requiring real-time visibility."

Data Overwhelm: "Existing systems presented too much information without prioritization, making it difficult to focus on critical decisions that impact patient care."

Manual Processes: "Teams wanted to upload existing scheduling files rather than re-inputting data, and needed clear visualization of schedule changes."

Time-Sensitive Planning: "Users needed to monitor and optimize schedules for the next two weeks while running scenario planning for better resource allocation."

üí° User Testing & Feedback

After creating the initial concepts, I conducted extensive user testing sessions with Division Managers, Booking Clerks, and Directors. The feedback was invaluable and revealed specific improvements needed to truly address the user needs we had documented in research.

Critical Improvements Identified

1. File Upload System

User Quote: "Information from so many resources that inputting would be too time-consuming"

Needed: Excel file import capability to eliminate manual data entry burden

2. Schedule Snapshot View

User Quote: "16 surgeons hard to visualize" - need complete overview at a glance

Needed: Condensed snapshot view with hover details showing surgeon assignments and bed usage

3. Customizable Dashboards

Finding: Different roles (Directors, Division Managers, Booking Clerks) need different metrics

Needed: Role-based KPI customization so each user sees relevant metrics for their decisions

4. Automated Conflict Detection

Pain Point: Booking clerks spending 2-3 hours daily manually checking for scheduling conflicts

Needed: Real-time conflict detection with visual alerts to prevent double-bookings

5. Better Visual Polish

Feedback: Color palette needed refinement, charts needed better labels and context

Needed: Design system with documented color tokens, improved data visualization quality

‚úÖ Great Feedback, Clear Direction: The user testing gave me exactly what I needed to create production-ready designs that would truly solve their problems. I was ready to iterate on the concepts and create detailed specifications for development.

‚ö†Ô∏è The Pilot: When Feedback Comes Too Late

The Problem: While I was conducting user testing and gathering this valuable feedback, the development team had already begun implementation based on my initial concepts. Due to tight timelines and resource constraints, development moved quickly‚Äîtoo quickly for the design iteration cycle.

By the time I compiled the feedback and was ready to redesign, the pilot was already substantially built. The company's financial pressures meant we couldn't pause development to incorporate the improvements. Additionally, I wasn't involved in the design handoff process or implementation reviews, so I couldn't guide the translation from concept to code.

The result: A pilot that was built from my initial concepts‚Äîwithout the critical improvements identified through user testing. The feedback that would have made the system truly user-centered sat documented but unimplemented.

The Gap: The pilot had different color saturation, tighter spacing, and lacked the critical features users requested: no file upload system, no schedule snapshots, no customizable KPIs, no automated conflict detection.

Key Lessons

  • Front-load user testing - Conduct testing BEFORE development starts, not during
  • Stay involved in handoff - Designer involvement during implementation preserves intent
  • Document early - Create design systems and specs upfront, not as an afterthought

‚è∏Ô∏è Project End: Shortly after the pilot launched, the company ran out of funding and ceased operations in 2022. The documented user feedback sat unimplemented‚Äîa missed opportunity to create the user-centered solution the research had pointed toward.
